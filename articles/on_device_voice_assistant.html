On-Device Voice Assistant — No Cloud, No Latency

April 2026 • 10 min read • Rohan Saxena •

Jarvis is a fully on-device voice assistant for iOS. Every component — wake word detection, speech recognition, language model inference, and speech synthesis — runs locally on the phone. No network calls, no cloud APIs, no data leaving the device. The entire pipeline fits in under 1 GB of RAM.

Full on-device loop: wake word, ASR, LLM, TTS — no internet required.

Why On-Device

Cloud-based voice assistants are fast and capable, but they depend on connectivity, introduce latency over the network hop, and require sending private audio to external servers. I wanted to see how far a fully local pipeline could go on a modern iPhone — no API keys, no backend, no internet required for the core loop.

The constraint forced careful decisions at every layer: which models to use, how to quantize them, how to schedule inference across threads, and how to keep the audio engine running continuously without gaps.

Wake Word Detection

The assistant activates on "Hey Jarvis" using openWakeWord, an open-source wake word engine. The detection pipeline is three ONNX models running in sequence: a mel spectrogram feature extractor, an embedding model, and a binary classifier tuned to the trigger phrase.

Wake word detection runs continuously in every state — idle, listening, processing, and speaking. During speech playback, it doubles as an interruption mechanism. If the user says "Hey Jarvis" while the assistant is talking, it immediately stops playback, clears the TTS queue, and transitions back to listening.

One subtlety: the audio samples fed to the wake word model must be in int16 range (-32768 to 32767) represented as float32, not the normalized [-1, 1] range that AVAudioEngine provides. Getting this wrong produces silent failures — the model runs but never triggers.

Streaming Speech Recognition

ASR uses NVIDIA's FastConformer RNN-T architecture, exported to ONNX and running directly through ONNX Runtime on the device. It's a streaming model — audio chunks are fed continuously, and transcription results arrive incrementally as the user speaks.

The system uses silence-based endpoint detection. A 1.2-second timer starts after the last ASR output. If no new speech arrives within that window, the accumulated transcript is finalized and passed to the language model. This avoids premature triggering on short pauses while keeping the interaction responsive.

  • Audio is captured at 16 kHz mono via AVAudioEngine's input tap
  • A ring buffer holds the last 2 seconds of audio to capture speech that starts before the state transition completes
  • ASR inference runs off the audio thread to avoid blocking real-time callbacks
  • Acoustic Echo Cancellation (AEC) via iOS voice processing prevents the mic from hearing the assistant's own playback

Local LLM Inference

The language model is Llama 3.2 1B, quantized to Q4_K_M (4-bit, 770 MB on disk), running through llama.cpp's C API. Inference happens on a background thread. Tokens stream out one at a time via an AsyncStream, and the pipeline begins TTS synthesis as soon as the first complete sentence is available.

The prompt template uses Llama 3's chat format with a system prompt that enforces concise, voice-friendly responses — short sentences, no markdown, no lists. Chat history is bounded to the last 10 messages to keep prompt length under the 2048-token context window.

Tool Calling: Web Search

The LLM supports a single tool: web search. When it encounters a question about current events or facts it's uncertain about, it outputs a [SEARCH: query] tag. The pipeline detects this mid-stream, cancels the current generation, executes a DuckDuckGo search, injects the results into the chat history, and restarts generation with the new context.

This is a two-pass loop. The first pass may produce a search tag. If it does, the second pass generates the final answer grounded in real search results. If the first pass completes without a search tag, it's used directly. The entire mechanism is transparent to the user — they just hear the answer.

On-Device Text-to-Speech

TTS uses Piper, a VITS-based neural speech synthesis model running through ONNX Runtime. The model takes phoneme IDs as input and produces raw float32 audio at 22050 Hz in a single forward pass. No vocoder, no mel spectrogram intermediate — end-to-end.

Text-to-phoneme conversion uses a 178,000-word pronunciation lexicon with a neural grapheme-to-phoneme (G2P) fallback for unknown words. One challenge was bridging between the lexicon's phoneme format and Piper's expected IPA — ligature characters like ʧ and ʤ needed to be decomposed into their component phonemes (t+ʃ, d+ʒ) for the model to produce correct output.

Synthesis is chunked at sentence boundaries. As the LLM streams tokens, complete sentences are dispatched to a dedicated TTS serial queue. Each sentence is synthesized independently and scheduled on an AVAudioPlayerNode for gapless playback. This means the user hears the first sentence while subsequent sentences are still being generated.

Audio Engine Architecture

The entire audio pipeline runs through a single AVAudioEngine instance that never stops between state transitions. Two AVAudioPlayerNodes handle output — one for filler sounds (greetings, acknowledgments) and one for TTS speech — allowing crossfading and independent volume control.

Voice processing (AEC) is enabled at the hardware level via setVoiceProcessingEnabled(true) on the input node. This is the same echo cancellation that FaceTime uses. Combined with the .voiceChat audio session mode, it effectively removes speaker output from the mic signal, preventing the ASR from transcribing the assistant's own voice.

Even with AEC, an additional software guard delays ASR activation until greeting playback completes — the delay is calculated from the actual WAV file duration plus a 500ms safety margin. The ring buffer is drained and discarded before ASR begins, ensuring no stale audio contaminates the transcript.

Interruption Handling

Interruption is first-class. The wake word detector runs in every pipeline state, so saying "Hey Jarvis" during speech playback triggers a full pipeline reset: TTS playback stops, the LLM generation task is cancelled, the TTS chunk queue is flushed, and ASR state is cleared. The assistant plays a short acknowledgment and transitions back to listening.

This requires careful concurrency management. The LLM runs on a background dispatch queue, TTS synthesis runs on a dedicated serial queue, and audio scheduling happens on AVAudioEngine's internal threads. Cancellation signals must propagate across all of these without deadlocking. Swift's Task cancellation and a shared atomic flag on the LLM handle this cleanly.

The Filler Sound System

To make the assistant feel responsive during the gap between the user finishing their sentence and the first TTS audio arriving, the system plays pre-generated filler sounds — short phrases like "Let me think about that" or "One second." These are synthesized once at first launch using the same Piper TTS engine and cached as WAV files on disk.

There are 12 categories of fillers (thinking, search start, search result, error, interruption acknowledgment, etc.), each with 3 variants randomly selected at runtime. They bridge the perceptual gap without adding actual latency — the LLM is already running in parallel.

Model Budget

Everything fits comfortably on a modern iPhone:

  • Wake word: 3 ONNX models, ~5 MB total
  • ASR: FastConformer RNN-T (int8), ~80 MB
  • LLM: Llama 3.2 1B Q4_K_M, ~770 MB (mmap'd, not fully resident)
  • TTS: Piper VITS, ~65 MB + 178K-word lexicon + neural G2P
  • Total disk: ~920 MB. Peak RAM: under 1 GB

The LLM uses mmap, so iOS can page it in and out as needed. The ASR and TTS models are kept fully resident in memory for instant inference.

What I Learned

Building this reinforced something I first encountered in my cloud-based voice system: the hardest part of voice AI is not the models — it's the orchestration. Silence detection, echo cancellation, interruption propagation, thread-safe queue management, audio scheduling, and state machine transitions matter more than which LLM you pick.

On-device adds another dimension: every model choice is a tradeoff between quality and memory. A 1B parameter model fits but won't match GPT-4. Int8 ASR is fast but less accurate than float32. 4-bit quantization saves RAM but introduces subtle generation artifacts. The art is finding the combination where the overall experience still feels natural.

The most satisfying moment was the first time the full loop worked in airplane mode — wake word, transcription, response, speech — all happening locally in under 3 seconds. No server, no internet, no API bill. Just the phone thinking out loud.
